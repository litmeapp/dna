Message Anatomy

Ironclad rule — applies to every message in both chat and terminal, regardless of sender.

1.
Attachments — 0..N media/files, horizontal carousel
optional
2.
Text — markdown or plain text body
optional
3.
Widget — 0..1 rich interactive card (link preview, changeset, activity, etc.)
optional, max 1
Rules
  • Order is fixed: attachments → text → widget. Top to bottom. Never rearranged.
  • Attachments are always a horizontal carousel above the text. Media thumbnails first, file pills after. No vertical stacking.
  • Text is markdown (rendered as prose) or plain text. One text block per message.
  • Widget is at most one per message. Link preview, changeset, activity card, artifact pill, map, dashboard — any rich interactive card. Placed below text.
  • Minimum content: a message must have at least one of the three parts. Empty messages are invalid.
  • Media-only bubble: when a peer/user message has a single media attachment and no text, the media fills the entire bubble edge-to-edge — no padding. The bubble becomes the media.
  • Chat agent: flat layout — attachments → text → widget all as siblings (not inside bubble). Media scrolls full width.
  • Chat peer/user: attachments + text inside bubble. Widget as sibling below bubble.
  • Terminal: inline flow — attach-row → text lines/prose → card.

Rules
  • Single media, no text: image fills the bubble. No padding. Bubble shape = image shape (clipped by border-radius). This is the "photo message" pattern.
  • Single media + text: image still edge-to-edge at top. Text below with horizontal padding. Bubble width driven by image aspect ratio, max 50%.
  • Multiple media: horizontal scroll. Each thumb is 70% of bubble width so 30% of the next item peeks — the visual cue that scrolling is available.
  • File pills: always vertical list, below media (if any), above text. Max 320px width. Rounded (radius10).
  • Agent: no bubble — media and files are flat siblings. Media carousel bleeds to full chat width. No max-width constraint.
  • Mixed media + files: media carousel first, file list second, text third. Always this order (anatomy rule).
  • Terminal geometry: terminal pills, thumbs, and option rows are square-cornered (0 radius). Chat uses rounded corners (radius10–radius12) concentric with bubbles. This split is intentional — terminal surfaces are utilitarian and grid-aligned; chat surfaces are organic and conversational.
Platform notes
  • iOS Single media: AsyncImage with .clipShape() matching bubble shape. Multiple: UICollectionView with paging-like scroll (70% width cells).
  • Android Single media: AsyncImage with Modifier.clip(). Multiple: LazyRow with Arrangement.spacedBy(4.dp), item width = 0.7f * maxWidth.
  • Web Single: width: 100% inside bubble. Multiple: flex-shrink: 0; width: 70% per thumb, overflow-x: auto on container.
  • All Peek pattern is critical — if the user can't see the next item, they won't know to scroll. 70% is the sweet spot: enough to show the current image, enough peek to hint at more.

Platform notes

When to use
  • Chat anatomy per message: each agent message has at most 1 widget. Folds are inline metadata, not widgets. Text + changeset = OK. Text + widget = OK. Text + changeset + widget = split into 2 messages.
  • Terminal flow: prompt → folds → steps → prose → changeset → artifact. All in one continuous stream. No message boundary — output flows naturally.
  • Peer in chat: media inside bubble (single = edge-to-edge). Peer in terminal: @julia username prefix.
  • Grouping: consecutive user capsules group (tight spacing, shared timestamp).
Platform notes
  • iOS Chat: UICollectionView with self-sizing cells. Terminal: NSAttributedString-backed scroll view.
  • Android Chat: LazyColumn with reverseLayout = true. Terminal: LazyColumn with auto-scroll.
  • Web Chat: flex column, overflow-y: auto. Terminal: .term-surface with scrollIntoView.
  • All Virtualize long sessions (>500 items). Input stays above keyboard on mobile.
