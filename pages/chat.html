When to use
  • .chat-msg--incoming/Agent responses — flat text, no bubble. The absence of a container separates AI from human messages visually.
  • .chat-msg--outgoing/User messages to agent. Blue bubble, right-aligned. Text is white with text shadow.

Platform notes

  • Web .chat-msg--outgoing { align-self: flex-end; flex-direction: row-reverse; } — flex handles right-alignment.
  • All Agent messages = flat text (no bubble). This is the key visual distinction between AI and human messages.

Bubble rules
  • Single line = always capsule. If content fits one line, add .chat-msg-bubble--capsule. No exceptions.
  • Standalone tails: Peer = top-left, User = bottom-right. Same chatTailRadius (6px) everywhere.
  • Grouped tails face inside. First → tail down only. Middle → tails up+down. Last → tail up only. Outside corners stay clean.
Grouping rules
  • Trigger — consecutive messages from the same sender, sent within 60 seconds.
  • .chat-msg--group-start/First in group. Tail faces inside (down). Outgoing: BR. Peer: BL (natural TL removed). Shows name, avatar, no timestamp.
  • .chat-msg--grouped/Middle messages. Tails on both sender-side corners (outgoing: TR+BR, peer: TL+BL). Tight spacing, hides name/avatar/timestamp.
  • .chat-msg--group-end/Last in group (add alongside .chat-msg--grouped). Tail faces inside (up). Outgoing: TR only (BR removed). Peer: TL only. Shows timestamp.
  • Tail rule — tails always face the interior of the group, linking adjacent messages. Outside corners (top of first, bottom of last) stay clean.
  • Capsules follow the same rules — same corner logic, pill radius instead of chatBubbleRadius.
Platform notes
  • iOS Detect single-line in layout pass: if intrinsicContentSize.height ≤ singleLineHeight, use Capsule() clip shape. Measure after text layout, not before.
  • iOS Grouping: compare message.timestamp deltas in the data source. If <60s and same sender, set isGrouped = true and hide supplementary views (avatar, name, time).
  • Android Capsule: use RoundedCornerShape(24.dp) pill radius — do NOT use CircleShape, which causes the same proportional clamping issue as CSS border-radius: 9999px when a tail corner is 6dp.
  • Android Grouping: implement with LazyColumn and determine item shape (standalone/start/middle/end) based on position in the data list. Same 60s + same-sender trigger.
  • Web Add .chat-msg--grouped to consecutive same-sender messages within 60s. CSS handles the tight spacing and hidden elements — no JS layout needed.

Platform notes

Why no focus ring
  • Singleton input/The composer is the only text input in the chat viewport. There is no ambiguity about which element has focus — a ring adds visual noise, not information.
  • Always visible/Unlike form inputs that share a page with many fields, the composer sits in a fixed bar at the bottom. Its position and role are always obvious.
  • Keyboard drives context/On mobile, the keyboard itself is the focus indicator. On desktop, the blinking caret inside the field confirms focus.
  • Not a blanket rule/This exception applies only to singleton always-visible inputs (chat composer, search bar in a search-only screen). All other inputs must show focus-visible.
Platform notes
  • iOS UITextField in the compose bar has no system focus ring by default. Keyboard avoidance via keyboardLayoutGuide is the only required affordance.
  • Android TextField in Compose: set focusedIndicatorColor = Color.Transparent via TextFieldDefaults.colors(). The soft keyboard is the focus indicator.
  • Web Apply .input-demo--chat to suppress the focus ring. Only use on singleton always-visible composer inputs — never on form fields.
  • All If the composer shares screen space with other inputs (e.g. inline editing + message typing), re-enable the focus ring to disambiguate.
