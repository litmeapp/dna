When to use
  • .window/The outermost terminal container. Use for any standalone console instance — command line, REPL, log viewer.
  • .window-titlebar/Window chrome with traffic lights and title. Omit in embedded/inline terminals where chrome is unnecessary.
  • .window-dot/Close (red), minimize (yellow), maximize (green). Decorative on Web, functional on native. Always render all three — partial sets look broken.
Platform notes
  • iOS Use NSWindow-style title bar for macOS Catalyst. On iOS proper, omit traffic lights — use system navigation chrome instead.
  • Android Title bar maps to a custom TopAppBar. Traffic lights are decorative only — use system back/close affordances.
  • Web .window-dot elements are purely visual. Do not attach click handlers unless implementing actual window management (electron, PWA).
  • All The terminal follows the system theme. Do not force dark mode — the warm neutral surface in light theme is intentional, honoring the Braun influence.

When to use
  • .term-prompt/The interactive input line. Always shows the prompt symbol ($) — it's the entry point to the terminal.
  • .term-prompt-symbol/Uses --tint because the prompt symbol is the terminal's primary call to action. The brand color signals "your turn."
  • .term-prompt-ghost/Autocomplete suggestion shown inline. Uses --quaternaryLabel so it never competes with actual input. Accept with Tab/Right arrow.

Platform notes
  • iOS Cursor: use CABasicAnimation with .timingFunction = CAMediaTimingFunction(name: .easeInEaseOut). Block cursor (filled rect), not line cursor — it's more visible on mono text.

  • Web @keyframes term-blink with steps(2) — sharp on/off, no fade. Respects prefers-reduced-motion (animation stops, cursor stays visible).
  • All Ghost text is hint-only — never commit it without explicit user action (Tab or right-arrow). Minimum 44pt tap target on the input area for mobile.

When to use
  • .term-line--stdout/Standard command output. Uses --secondaryLabel — one step below the user's input, creating a clear visual hierarchy: input (bold, full contrast) → output (softer, secondary).
  • .term-line--stdin/Echoed user commands in history. Full --label contrast to distinguish "what you typed" from "what the system said."
  • .term-line--muted/Metadata, counters, timestamps. Tertiary contrast — visible but doesn't compete with actual content.
Platform notes
  • iOS Use Text views in a LazyVStack(alignment: .leading, spacing: 0). Line height: .lineSpacing(14 * 0.7) to match 1.7× at 14pt.
  • Android LazyColumn with Text items. Set lineHeight = 23.8.sp (14 × 1.7). Use fontFamily = ibmPlexMono.
  • Web .term-line uses white-space: pre-wrap to preserve indentation while allowing wrapping. word-break: break-all prevents horizontal overflow on long unbroken strings (e.g., URLs, hashes).
  • All Hierarchy: user input (--label) stands above output (--secondaryLabel). This is the terminal's core contrast rule — "who said what" is always clear.

// Token resolver — maps CSS custom properties to computed values
function resolveToken(name, el) {
const style = getComputedStyle(el || document.documentElement);
const value = style.getPropertyValue('--' + name).trim();
return value || 'undefined';
}
// Usage
const tint = resolveToken('tint'); // → "#E86420"

When to use
  • Syntax highlighting maps to existing system colors — no new tokens. This keeps the terminal palette in sync with the rest of the design system.
  • .term-kw/--systemPink — reserved words. Pink is distinctive and unused in semantic UI, so it reads unambiguously as "language construct."
  • .term-str/--systemGreen — universal convention across every major editor. Green = literal data.
  • .term-flag/--tint — CLI flags modify behavior, same semantic as the brand action color.
  • .term-comment/--tertiaryLabel — comments recede. They're metadata, not content.
Platform notes
  • iOS Use AttributedString with .foregroundColor per token range. Match the system color tokens: .systemPink, .systemGreen, .systemBlue, etc.
  • Android Use AnnotatedString with SpanStyle(color = ...) per token range. Map to Material Color values matching the system color hex values.
  • Web Wrap each token in a <span class="term-*">. CSS custom properties handle theme switching automatically.
  • All Zero new tokens. Every highlight color is an existing system color that already adapts to light/dark themes.

When to use
  • .term-ansi-*/Use when rendering ANSI escape code output (e.g., from external CLI tools, git diff, test runners). Maps standard terminal colors to FORK's warm palette.
  • Bright variants (8–15) share tokens with standard (0–7) because FORK's system colors already shift between themes. Adding separate bright tokens would violate the Token Economy.
  • ANSI cyan maps to --tintTeal — the closest warm-palette equivalent. No --systemCyan exists because FORK's color system prioritizes warmth over spectral completeness.
Platform notes
  • iOS Parse ANSI escape sequences with a custom AttributedString builder. Map \033[31m (red) → .systemRed, etc. Strip escape codes from accessibility labels.
  • Android Use AnnotatedString with SpanStyle for each ANSI code range. Consider a reusable AnsiParser composable that converts raw output to styled text.
  • Web Parse ANSI codes to <span class="term-ansi-*">. Libraries like ansi-to-html can be configured with custom class names.
  • All ANSI 256-color and truecolor (24-bit) are out of scope. FORK terminals use the 16-color palette exclusively — it covers all standard terminal output.

Session anatomy
  • Command → Output → Status is the fundamental rhythm. Every command produces output, every output ends with a status line (success/error). This consistency is non-negotiable.
  • Visual hierarchy flows down: prompt (tint-colored symbol) → user input (full contrast) → output (secondary) → metadata (tertiary). Each level recedes, directing attention to the user's commands.
  • Blocks interrupt the stream — diffs and tables get their own contained surface (tertiary background inside secondary). This creates a visual "well" that signals structured content.
  • Progress is inline — progress bars sit in the output stream, not in overlays or modals. The terminal is a single, continuous narrative.
Platform notes
  • iOS Session history: use a NSAttributedString-backed UITextView (non-editable) for output + a separate UITextField for input. Auto-scroll to bottom on new output.
  • Android Session: LazyColumn for history + TextField at bottom. Use LaunchedEffect to scroll to latest item. Match the ibmPlexMono family across all text elements.
  • Web Session is a scrollable .term-surface div. New output appends to the DOM. Use scrollIntoView({ behavior: 'smooth' }) for auto-scroll. Consider ResizeObserver to maintain scroll-to-bottom on container resize.
  • All The session is a living document — it grows over time. Virtualize long sessions (>500 lines) to maintain performance. On mobile, ensure the input area stays above the keyboard.
