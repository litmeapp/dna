When to use
  • .window/The outermost terminal container. Use for any standalone console instance — command line, REPL, log viewer.
  • .window-titlebar/Window chrome with traffic lights and title. Omit in embedded/inline terminals where chrome is unnecessary.
  • .window-dot/Close (red), minimize (yellow), maximize (green). Decorative on Web, functional on native. Always render all three — partial sets look broken.

  • All The terminal follows the system theme. Do not force dark mode — the warm neutral surface in light theme is intentional, honoring the Braun influence.

When to use
  • .term-prompt/The interactive input line. Always shows the prompt symbol ($) — it's the entry point to the terminal.

const style = getComputedStyle(el || document.documentElement);
const value = style.getPropertyValue('--' + name).trim();
return value || 'undefined';
}

When to use
  • Syntax highlighting maps to existing system colors — no new tokens. This keeps the terminal palette in sync with the rest of the design system.
  • .term-kw/--systemPink — reserved words. Pink is distinctive and unused in semantic UI, so it reads unambiguously as "language construct."
  • .term-str/--systemGreen — universal convention across every major editor. Green = literal data.
  • .term-flag/--tint — CLI flags modify behavior, same semantic as the brand action color.
  • .term-comment/--tertiaryLabel — comments recede. They're metadata, not content.
Platform notes
  • iOS Use AttributedString with .foregroundColor per token range. Match the system color tokens: .systemPink, .systemGreen, .systemBlue, etc.
  • Android Use AnnotatedString with SpanStyle(color = ...) per token range. Map to Material Color values matching the system color hex values.
  • Web Wrap each token in a <span class="term-*">. CSS custom properties handle theme switching automatically.
  • All Zero new tokens. Every highlight color is an existing system color that already adapts to light/dark themes.

Spacing Tokens

The spacing system uses a geometric scale based on multiples of 4px. Each token maps to a specific spatial relationship in the layout.

Scale

  • space4 — Tight grouping, inline elements
  • space8 — Default gap between related items
  • space16 — Section padding, card insets
  • space24 — Group separation
  • space48 — Section dividers

Rules

  1. Related items use smaller tokens (space4–space8)
  2. Unrelated items use larger tokens (space24–space48)
  3. Never skip more than one step in the scale
Whitespace is content — generous padding signals confidence.

See the layout reference for the full token table.

Session anatomy
  • Command → Output → Status is the fundamental rhythm. Every command produces output, every output ends with a status line (success/error). This consistency is non-negotiable.
  • Visual hierarchy flows down: prompt (tint-colored symbol) → user input (full contrast) → output (secondary) → metadata (tertiary). Each level recedes, directing attention to the user's commands.
  • Blocks interrupt the stream — diffs and tables get their own contained surface (tertiary background inside secondary). This creates a visual "well" that signals structured content.
  • Progress is inline — progress bars sit in the output stream, not in overlays or modals. The terminal is a single, continuous narrative.
  • Agent session — the prompt $ marks user input, everything between prompts is the assistant's response. No extra boundary marker needed — the tint-colored symbol is the visual anchor. Tool calls fold above the prose response.
Platform notes
  • iOS Session history: use a NSAttributedString-backed UITextView (non-editable) for output + a separate UITextField for input. Auto-scroll to bottom on new output.
  • Android Session: LazyColumn for history + TextField at bottom. Use LaunchedEffect to scroll to latest item. Match the ibmPlexMono family across all text elements.
  • Web Session is a scrollable .term-surface div. New output appends to the DOM. Use scrollIntoView({ behavior: 'smooth' }) for auto-scroll. Consider ResizeObserver to maintain scroll-to-bottom on container resize.
  • All The session is a living document — it grows over time. Virtualize long sessions (>500 lines) to maintain performance. On mobile, ensure the input area stays above the keyboard.
